## Spring Cloud Gateway
백엔드 서비스들의 구성이 변하는 등과 같은 이슈가 있을때, 소프트웨어 레벨의 Gateway 가 없는 상태에서는 백엔드 구성의 변경사항을 여러 서비스에 통지하고 frontend 에도 이와 같은 내용을 반영해야 합니다. 이 경우 배포 역시 빅뱅과 같은 배포가 될수 있습니다. 또는 API 버전별 호환이 안되는 심각한 상황이 발생할 수도 있습니다.<br/> 

그런데, 소프트웨어 레벨에서 백엔드 팀에서 별도로 Gateway를 제공할 경우 Gateway 내에서 route 를 변경하는 등의 대응을 통해 유연하게 대응이 가능합니다.<br/>

소프트웨어 레벨의 Gateway 를 도입하지 않은 점이 계속 마음에 걸렸으며, 이런 이유로 Season2 개발 시에는 Spring Cloud Gateway 를 도입할 예정입니다.<br/>
<br/>


## 회원 탈퇴 기능, 비밀번호 변경 기능
급하게 하다보니... 그리고 인프라 배포 스크립트에 집착하다보니... 회원 탈퇴기능까지는 신경을 쓰지 못했습니다. Season 2 개발시에는 `회원 탈퇴기능`, `비밀번호 변경 기능` 을 반드시 추가할 계획입니다.<br/>
<br/>


## authentication-svc, member-svc 분리 검토
season 2 개발 시에는 authentication-svc 와 member-svc 로 멤버조회/관리와 인증관리를 분리할지 검토 해볼 예정입니다. 인증 도메인의 경우 인증 수단이 다양해진다거나 인증의 트래픽을 따로 관리하고 싶은 케이스가 있을 수 있고 인증 쪽의 자주 발생하는 변경사항이 `member` 라는 영역에 지나치게 영향을 끼치는게 올바른 것일지를 생각해보고 분리를 결정해보기로 했습니다.<br/>
<br/>


## OAuth 2.0 지원
Season2 에서는 구글 로그인 등에 대해 지원되도록 구현예정입니다. OAuth 2.0 을 지원하고 인증수단을 여러가지를 지원하게 될 경우 `authentication-svc` 를 따로 분리하게되지 않을까 싶습니다.<br/>
