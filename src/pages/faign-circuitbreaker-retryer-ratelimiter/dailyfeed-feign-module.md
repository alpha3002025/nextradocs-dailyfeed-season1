# dailyfeed-feign 모듈

생각해보면 feign 은 플랫폼 내에서 각각의 역할을 하는 서비스간 통신을 할때 사용되는 '통신 인터페이스'의 역할을 하게 됩니다. 만약 feign 을 개별 서비스마다 별도로 정의하게 된다면, 개발 초기 단계에서는 크게 문제가 되지는 않지만, 통신을 수행해야 할 서비스가 많아지거나 feign 통신을 수행할 엔드포인트의 종류가 많아질 경우에는 이야기가 달라지게 됩니다. 예를 들면 서비스의 종류가 많아질 경우 feign 통신 시에 적용할 공통적인 규칙이나 헤더, 특정 응답에 대한 Exception 처리 등을 규칙화해야 하는 상황 등 다양한 종류의 상황들이 발생하게 됩니다.<br/>
<br/>

**e.g. feign 기능/엔드포인트를 서비스 별로 개별 정의해서 사용시**<br/>
![](./img/the-dailyfeed-feign-module/20251231-16-12-01-1.png)

<br/>

`feign` 기능/엔드포인트를 서비스 별로 개별 정의해서 사용시에는 통신 시 발생하는 오류,헤더 처리에 대한 변경사항 발생시 각각의 개별 서비스를 수정해야 하기에 서비스의 기능이 다양해질 수록 작업이 점점 복잡해지고 손이 많이가게 됩니다.<br/>
<br/>

**e.g. feign 기능/엔드포인트의 코드를 모듈로 추출+정의 후 통신인터페이스 역할을 수행하도록 정의**<br/>
![](./img/the-dailyfeed-feign-module/20251231-16-12-21-1.png)

<br/>

feign 기능/엔드포인트의 코드 들을 `dailyfeed-feign`라는 모듈로 분리해서 각각의 서비스에서 서브모듈을 import 해서 사용하게 되는 경우입니다. 이 경우 feign 통신 시에 적용할 공통적인 규칙, 헤더, 특정응답에 대한 처리 등이 하나의 규칙화되고, 각각의 서비스에 대한 엔드포인트 역시 인터페이스 처럼 사용되기 때문에 개발 및 관리에 조금 더 용이해지게 됩니다.<br/>
